# OpenSciFlow v0.2 RFC outline

v0.2 should turn the current reviewable metadata drafts into a narrow executable contract for local-first agents.

This outline is not the RFC itself. It defines what the RFC must answer before v0.2 can be proposed for public review.

## Core question

What is the smallest protocol surface that lets a local agent safely run reviewed AI for Science tools while recording enough evidence for citation, audit, and reproduction?

## Required decisions

### 1. Command templates

- Which placeholders are allowed?
- How are input and output references typed?
- How does the agent render commands without invoking a shell parser?
- Which shell features are forbidden in manifest commands?
- When should wrapper scripts be allowed?

### 2. Environment readiness

- How should Conda, Docker, Apptainer, system packages, and Slurm readiness be represented?
- Which fields are required before dry-run execution?
- How should local package versions be recorded?
- How should failed readiness checks be represented?

### 3. Model weights and sample data

- Which source, license, checksum, version, and retrieval fields are mandatory?
- How should unavailable checksums be justified?
- How should a manifest disclose remote services or external downloads?
- When must the agent stop and ask for user approval?

### 4. Workflow artifact handoff

- Should workflow steps declare `consumes` and `produces`?
- How should intermediate artifacts be named?
- How should final outputs be linked to report templates?
- How should optional branches and fallback tools be represented?

### 5. Run records

- Which run-record fields are mandatory for R5?
- How should step-level logs and artifact hashes be represented?
- How should citations and limitations propagate from plugins to reports?
- What is the minimum valid run record for a dry run, smoke test, and full workflow?

### 6. Readiness levels

- Which evidence moves a manifest from R2 to R3?
- Which evidence moves a manifest from R3 to R4?
- Which evidence is required before a workflow can claim R6?
- Which readiness labels should be visible to users before execution?

## Non-goals for v0.2

- Defining a full workflow engine.
- Replacing Nextflow, Snakemake, CWL, Galaxy, AiiDA, Parsl, Slurm, Conda, Spack, or Apptainer.
- Certifying scientific correctness.
- Supporting arbitrary LLM-generated shell commands.
- Building a hosted execution platform.

## Proposed v0.2 deliverables

1. Updated plugin manifest schema.
2. Updated workflow template schema.
3. Updated run-record schema.
4. Command-template renderer test cases.
5. Artifact handoff validator.
6. R3 dry-run evidence for at least one plugin.
7. One BioPilot sample run record that validates against the schema.
8. A public change log explaining incompatible changes from v0.1.

## Minimum acceptance bar

v0.2 should not be announced until:

- one plugin reaches documented R3;
- one workflow passes schema, DAG, and artifact handoff checks;
- one sample run record validates;
- the local-agent refusal rules are documented;
- known limitations remain visible in the generated report.
