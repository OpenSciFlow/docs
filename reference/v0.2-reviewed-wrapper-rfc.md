# v0.2 Reviewed Wrapper RFC Draft

This draft defines the question OpenSciFlow v0.2 must answer before agents can use wrapper scripts for local, container, or Slurm execution.

It is not a stable standard.

## Problem

Many scientific tools are not safely represented as one flat command.

Common examples include:

- Slurm job scripts;
- module or environment setup;
- Apptainer/Docker launchers;
- multi-step preprocessing wrappers;
- site-specific filesystem staging;
- tools that require stdout/stderr handling and scheduler metadata.

OpenSciFlow v0.1 forbids arbitrary shell composition inside manifest command strings. That keeps examples inspectable, but it leaves a gap: real HPC workflows often need reviewed wrapper scripts.

## Proposed Principle

A wrapper script may be used only when it is a reviewed protocol artifact.

An agent must not invent a wrapper script during execution and then treat it as trusted.

## Proposed Fields

For plugin manifests:

```yaml
execution:
  slurm:
    supported: true
    submit_command: "sbatch {run_directory}/job.sbatch"
    reviewed_wrapper:
      path: "wrappers/example.sbatch"
      review_status: "reviewed"
      reviewed_by: "OpenSciFlow maintainers"
      allowed_arguments:
        - run_directory
        - account
        - partition
        - time
        - cpus_per_task
        - mem
```

For skill execution requests:

```json
{
  "command_source": "reviewed-wrapper",
  "command_template_id": "execution.slurm.submit_command",
  "rendered_command": "sbatch runs/example/job.sbatch",
  "reviewed_wrapper": {
    "path": "wrappers/example.sbatch",
    "review_status": "reviewed",
    "reviewed_by": "OpenSciFlow maintainers",
    "allowed_arguments": ["run_directory", "account", "partition"]
  }
}
```

## Required Checks

Before rendering a wrapper, the agent should check:

- wrapper path is declared;
- review status is `reviewed`;
- reviewer or reviewing project is declared;
- all fillable arguments are listed;
- no undeclared argument is filled;
- rendered wrapper is written inside the approved run directory;
- rendered submission command uses only allowed placeholders;
- stdout and stderr paths are declared;
- citation, license, limitations, and run-record behavior are preserved;
- user approval is required before submission or resource-consuming execution.

## Refusal Cases

The agent should refuse execution when:

- wrapper metadata is missing;
- wrapper review status is absent or not `reviewed`;
- the user asks the agent to create a new unreviewed wrapper;
- scheduler account, partition, time limit, module, or container field is required but unknown;
- wrapper would write outside the approved run directory;
- wrapper downloads model weights or data without source, license, checksum, and approval;
- wrapper suppresses errors without preserving logs;
- wrapper turns computational output into clinical, drug-efficacy, or experimental claims.

## Run-Record Requirements

For wrapper-based execution, the run record should include:

- rendered submission or launcher command;
- wrapper path;
- wrapper hash if available;
- rendered wrapper path;
- scheduler job id if submitted;
- stdout and stderr log paths;
- return code or scheduler final state;
- modules, container image, or environment identifier;
- input and output hashes;
- citations, licenses, limitations, and warnings.

## Open Questions

- Should `review_status` be an enum such as `draft`, `reviewed`, `deprecated`?
- Should wrapper hashes be mandatory before R5 execution?
- Should Slurm fields use Slurm-native names such as `cpus_per_task` and `mem`, or normalized names such as `cpu_cores` and `memory_gb`?
- Should `submit_command` live in the plugin manifest, workflow template, or both?
- Should `reviewed_by` allow a project name, a GitHub handle, a signed commit, or all three?
- How should site-local private wrappers be referenced without leaking cluster paths?

## Minimum Acceptance Bar

This RFC section is ready for v0.2 only when:

- one plugin manifest includes reviewed-wrapper metadata;
- one workflow template consumes that metadata;
- one skill execution request validates against reviewed-wrapper schema rules;
- one refusal test blocks missing wrapper metadata;
- one issue has been reviewed by someone with real HPC or Slurm experience.
